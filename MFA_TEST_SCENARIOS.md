# MFA & Rescue Links — сценарии проверки

Сценарии для ручной проверки функциональности 2FA и rescue-ссылок в плагине Limit Login Attempts Reloaded.

---

## Предусловия

- Плагин активирован, HTTPS включён, OpenSSL доступен.
- В настройках LLAR → вкладка MFA: 2FA включено, для роли включена MFA, сгенерированы rescue-коды (есть хотя бы одна сохранённая rescue-ссылка).

---

## 1. Успешное использование rescue-ссылки

**Шаги:**

1. Войти на сайт под пользователем с MFA (логин + пароль + при необходимости код 2FA).
2. В настройках MFA скопировать одну rescue-ссылку (или взять из ранее сохранённого PDF/списка).
3. Открыть ссылку в браузере (желательно в режиме инкогнито или в другом браузере, чтобы не было активной сессии).
4. Убедиться, что произошёл редирект на страницу входа (`wp-login.php` или WooCommerce login) с параметром `llar_mfa_disabled=1`.
5. На странице входа должно отображаться сообщение в духе: «Multi-factor authentication has been temporarily disabled. Please try again later.» (без указания длительности в 1 час).
6. Войти логином и паролем без запроса 2FA — вход должен быть успешным.

**Ожидаемый результат:** MFA временно отключён, вход без 2FA возможен, сообщение абстрактное (без «1 hour»).

---

## 2. Повторное использование той же rescue-ссылки (double-spend)

**Шаги:**

1. Использовать одну и ту же rescue-ссылку дважды подряд (в двух вкладках или с небольшой задержкой).
2. Первый запрос: редирект на логин, как в сценарии 1.
3. Второй запрос по той же ссылке: страница с ошибкой (403), неинформативное сообщение (например «An error occurred.»), без редиректа на логин.

**Ожидаемый результат:** Ссылка одноразовая; второй запрос получает 403, повторного «списания» нет.

---

## 3. Cooldown rescue endpoint (ограничение по частоте)

**Шаги:**

1. Успешно использовать одну rescue-ссылку (редирект на логин).
2. В течение минуты (по умолчанию `LLA_MFA_RESCUE_USE_COOLDOWN`) открыть другую валидную rescue-ссылку.
3. Должен вернуться ответ 429 с сообщением вроде «Too many requests.» (без указания, когда можно повторить).

**Ожидаемый результат:** Вторая ссылка в период cooldown не обрабатывается, возвращается 429, текст не раскрывает тайминги.

---

## 4. Невалидный или истёкший rescue (hash_id / ссылка)

**Шаги:**

1. Открыть URL вида `https://example.com/?llar_rescue=invalid` (короткий или не hex).
2. Открыть URL с `llar_rescue` из 64 hex-символов, но не соответствующий ни одной выданной ссылке (или с истёкшим transient).
3. В обоих случаях: ответ 403, неинформативное сообщение об ошибке.

**Ожидаемый результат:** 403, без деталей (нет «invalid hash», «expired» и т.п. в пользовательском сообщении).

---

## 5. Состояние после истечения временного отключения (1 час)

**Шаги:**

1. Использовать rescue-ссылку, войти без 2FA.
2. Перейти в настройки LLAR → MFA.
3. Чекбокс «Enable 2FA» должен оставаться включённым; при необходимости отображается индикация временного отключения.
4. После истечения transient (по умолчанию 1 час, можно уменьшить через `LLA_MFA_DISABLE_DURATION` для теста): обновить страницу настроек MFA.
5. Чекбокс «Enable 2FA» по-прежнему включён, MFA снова требуется при входе (без повторного включения 2FA вручную).

**Ожидаемый результат:** После истечения времени MFA автоматически снова активен, чекбокс не сбрасывается, явно включать 2FA заново не нужно.

---

## 6. Явное отключение 2FA администратором

**Шаги:**

1. В настройках MFA снять галочку «Enable 2FA» и сохранить.
2. Убедиться, что при входе 2FA не запрашивается.
3. (Опционально) Если перед этим использовалась rescue-ссылка и transient ещё активен — после истечения transient 2FA не должен включиться сам: чекбокс остаётся выключенным.

**Ожидаемый результат:** Решение администратора «выключить 2FA» сохраняется; истечение rescue transient не включает 2FA обратно.

---

## 7. Валидация hash_id (длина и формат)

**Проверка на стороне сервера (например, через логи или тест):**

- `llar_rescue` длиной не 64 символа → 403.
- `llar_rescue` с не hex-символами (при длине 64) → 403.
- Корректный 64-символьный hex, но неизвестный/уже потреблённый → 403.

Используется константа длины кода (например `LLA_MFA_CODE_LENGTH` / `MfaConstants::CODE_LENGTH`), не захардкоженное 64.

---

## 8. Очистка транзиентов и конфигурации

**Проверка (при наличии доступа к БД или отладочных средств):**

- Ежедневная очистка rescue-транзиентов использует префикс из `LLA_MFA_TRANSIENT_RESCUE_PREFIX` (по умолчанию `llar_mfa_rescue_`).
- При отключении MFA через настройки удаляются rescue-коды и связанные транзиенты (в т.ч. по тому же префиксу).
- Опция `mfa_rescue_pending_links` в конфиге не используется (удалена); rescue-данные хранятся в отдельных транзиентах по `hash_id`.

---

## 9. Сообщения и безопасность

- На странице входа после редиректа по rescue: только абстрактное сообщение («temporarily disabled», «Please try again later»), без указания длительности (1 час).
- При 429 (cooldown): сообщение не раскрывает, через сколько секунд/минут можно повторить.
- При 403/500 в rescue endpoint: единое неинформативное сообщение об ошибке, без деталей реализации.

---

## Краткий чеклист

| # | Сценарий | Ключевая проверка |
|---|----------|-------------------|
| 1 | Успешный rescue | Редирект на логин, сообщение без «1 hour», вход без 2FA |
| 2 | Двойное использование ссылки | Второй запрос — 403 |
| 3 | Cooldown | Вторая ссылка в течение минуты — 429 |
| 4 | Невалидный/истёкший rescue | 403, неинформативное сообщение |
| 5 | После истечения transient | MFA снова включён без действий пользователя |
| 6 | Админ выключил 2FA | После истечения transient 2FA остаётся выключенным |
| 7 | Валидация hash_id | Длина/формат через константу, 403 при неверных данных |
| 8 | Очистка | Префикс из константы, нет `mfa_rescue_pending_links` |
| 9 | Тексты | Нет раскрытия длительностей и внутренних деталей |
